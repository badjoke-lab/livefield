# Twitch Livefield ディレクトリ構成書 v1.2 改訂版

## 1. 文書の位置づけ

本書は、**Twitch Livefield のリポジトリ構成・責務分離・将来拡張時の配置ルール** を定める。
優先順位は以下とする。

1. **Twitch Livefield 共通仕様書**
2. **Twitch Livefield サイト全体構成書**
3. **本書（ディレクトリ構成書）**
4. **Heatmap / Day Flow / Battle Lines 各機能仕様書**
5. 実装タスク

本書は「どのフォルダに何を置くか」を決める文書であり、UI の細かな仕様や API の返却値詳細は個別仕様書で扱う。
ただし、**どの層を分けるべきか、何を shared に寄せるべきか、いつ rename するか** は本書で固定する。

---

## 2. 本書の目的

本書の目的は以下とする。

* Twitch-only MVP を壊さずに維持する
* 将来の Kick 追加を見据えた 1リポ構成を先に固定する
* Shell / Renderer / API / Collector / Storage の境界を崩さない
* 1つの変更が全体破壊に直結しない構造にする
* docs / specs / code / fixtures / tests の置き場所を明確にする

---

## 3. 先に結論

Livefield は、**1リポ・2サイト・shared 共通・collector 分離** を最終形とする。

理由は以下である。

* 現仕様が **Twitch-only MVP を先に完成** させつつ、将来の Kick 追加は同じ考え方を踏襲するとしている
* サイト全体構成が **Shell / Renderer / API / Collector-Storage** の分離を前提にしている
* collector の失敗や provider 差分を UI 側に漏らしにくくする必要がある
* activity / momentum / payload contract の共通化余地が大きい

---

## 4. 構成原則

ディレクトリ構成は以下の原則で固定する。

### 4.1 公開単位を分ける

* Twitch 用公開サイト
* Kick 用公開サイト

は、同一 repo の中でも **公開単位としては分ける**。

### 4.2 collector を分ける

* Twitch collector
* Kick collector

は完全に分ける。
collector 障害の切り分けを優先し、`if (platform === ...)` を worker 1本の内部に増殖させない。

### 4.3 shared を厚くする

以下は shared に寄せてよい。

* 型
* 時刻ユーティリティ
* 正規化
* payload schema
* momentum 算出
* provider-independent activity の扱い
* Battle Lines 向け pair scoring
* 共通 copy / config / limits

### 4.4 docs と code を分ける

* docs は人間向け説明
* specs は仕様正本
* code は実装
* fixtures は demo / test 用
* tests は shared / integration / smoke

として分ける。

### 4.5 今の Twitch-only を壊さない

Kick 追加のための完成形を定めるが、**rename のタイミングは Kick scaffold 開始時** とする。
Twitch-only 完成前に repo 全体を大きく崩さない。

## 6. この完成形の考え方

この完成形での考え方は単純である。

### 6.1 `apps/web-twitch` と `apps/web-kick`

どちらも **`/ /heatmap /day-flow /battle-lines /method /about /status` の7ページだけ** を持つ公開単位とする。
現仕様では正式公開範囲はこの7ページで、クロスサイト対応は後回しなので、**1つの site の中で Twitch と Kick を混在させない** のが正しい。

### 6.2 `workers/collector-twitch` と `workers/collector-kick`

完全に分ける。
理由は、全体構成が **Collector / Storage 層を描画や Shell と分離** する前提で、しかも **collector の失敗を隠蔽しないこと** が必須だからである。
1本の worker に `if (platform===...)` を増やすと、障害時の切り分けが悪くなる。

### 6.3 `packages/shared`

かなり厚くしてよい。
Heatmap の `momentum`、provider-independent な `activity`、Day Flow の `viewer-minutes`、Battle Lines の `pairScore` 系は **shared 定数・shared scoring・shared payload** に寄せるべきである。
仕様でも activity は provider 実装差を許しつつ、UI と内部指標は共通化する前提である。

### 6.4 `db`

**repo 共通スキーマとプラットフォーム別保存** を併用する。
1つの D1 に全部混ぜてもよいが、運用上は **Twitch 用 D1** と **Kick 用 D1** を分ける前提で、migration は分けておく方が安全である。
共通仕様でも無料運営前提・取得不足時は `partial` 明示が必要なので、片方の collector 問題で両方を汚さない構造が向いている。

### 6.5 `tests`

**shared / twitch / kick** の3層に分ける。
特に Battle Lines は recommended/custom 状態や latest payload の扱いが重いので、shared unit と platform integration を分けた方が壊れにくい。

## 7. 現在の Twitch-only 実装との関係

本書は最終完成形を定めるが、**今すぐ全 rename を強制する文書ではない。**

### 7.1 いまの許容状態

Twitch-only MVP の進行中は、以下のような一時状態を許容する。

* `apps/web`
* `workers/collector`
* `docs/*.md`

のような単一サイト前提の配置

### 7.2 rename の実行条件

以下のいずれかに入った時点で rename を開始する。

* Kick 用 scaffold を実際に作り始める
* deploy workflow を Twitch / Kick に分ける
* docs / specs を Twitch と Kick で複製し始める
* collector を provider 単位で分ける実作業に入る

### 7.3 先に repo を崩しすぎない

Twitch-only MVP 完了前に、rename そのものを目的化して repo を大きく崩すのは禁止する。
まず本番で動く MVP を維持し、その後に rename する。

## 8. 今すぐやるべき rename と移動

完成形に合わせて、最初にやる rename と移動はこれで固定する。

```text

workers/collector       -> workers/collector-twitch
docs/*.md               -> docs/twitch/*.md
specs/*.md              -> specs/twitch/*.md
db/migrations           -> db/twitch/migrations
```

この順なら、いまの **Twitch-only MVP の一貫性** を壊さずに、将来の Kick 拡張原則にも乗れる。

## 10. `workers/` 配下のルール

### 10.1 `collector-twitch`

* Twitch の収集・保存・status 集計を担当する
* provider 固有 API 実装は `providers/twitch` に閉じ込める
* Heatmap / Day Flow / Battle Lines の UI ロジックを持ち込まない

### 10.2 `collector-kick`

* Kick 追加時に同じ責務分離を守る
* Twitch collector との shared 化は `packages/shared` を通じて行う
* 片方の障害が片方の worker を巻き込まないことを優先する

## 16. Heatmap production treemap 正式採用を踏まえた補足

## 18. Heatmap production treemap 採用に伴う配置追記事項

本節は、Heatmap 正式版を production treemap として採用したことに伴う、配置上の追記事項を定める。

### 18.1 用語

* ページ名は Heatmap
* 方式名は treemap
* 各矩形は tile

ディレクトリやモジュール名では、公開実装の主役を **treemap** として明示し、`tile` は個別要素や UI 補助の文脈に限って用いてよい。

### 18.2 treemap layout と bubble の分離

* production renderer は `renderers/heatmap/treemap/` に置く
* 旧 bubble や比較用 renderer は `renderers/heatmap/bubble/` または `pages/lab/heatmap-bubble/` に隔離する
* `/heatmap` 本番ページに renderer 比較コードを混在させない

### 18.3 interactions の分離

Heatmap treemap では、以下を `interactions/` で分けてよい。

* zoom
* pan / drag
* selection
* external link handling
* future grouping drill-down

これにより、layout / visual / interaction の責務を混在させない。

### 18.4 grouping を見据えた構造

grouping は MVP 後でもよいが、将来の追加を見据えて以下を許容する。

* `renderers/heatmap/treemap/grouping.ts`
* `pages/heatmap/state/grouping.ts`
* `packages/shared/types/heatmap-grouping.ts`

ただし Twitch-only MVP 完了前に空実装を大量追加することは目的化しない。

### 18.5 zoom を早期導入対象とする理由

Heatmap treemap は、viewers に比例した面積を素直に維持することが優先である。
小タイルを見やすくするために本来の size 比例を崩すのではなく、**zoom を正規機能として導入する** ほうが構造上も正しい。
そのため zoom は lab ではなく production 側の interaction として扱ってよい。

### 18.6 stream link 導線の配置

* tile 本体の選択と外部リンク導線は競合しやすいため、interaction 層で明示的に分離する
* 詳細パネル側の `Open stream` は pages/heatmap または components に置く
* 配信リンク判定を renderer 深部に埋め込みすぎない

### 18.7 TradingView 系 treemap を参考にする際の配置原則

参考にするのは以下のような一般原則である。

* dense packing
* size / color の分離
* detail layering
* zoom
* future grouping
