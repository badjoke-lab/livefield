# TwitCasting Livefield サイト全体構成書 v1.0

## 1. 文書の位置づけ

本書は、TwitCasting Livefield の**サイト全体の構造・ページ導線・データの流れ・実装境界**を定める。
優先順位は以下とする。

1. **TwitCasting Livefield 共通仕様書**
2. **本書（サイト全体構成書）**
3. **Heatmap / Day Flow / Battle Lines 各機能仕様書**
4. 実装タスク

---

## 2. サイト全体の目的

TwitCasting Livefield は、TwitCasting の配信状況を以下の3つの視点で観測するサイトとする。

* **Heatmap**
  今この瞬間の勢力図
* **Day Flow**
  1日の支配率と総量の流れ
* **Battle Lines**
  個別比較と逆転・急伸の分析

この3機能を、単発ページではなく**1つの観測サイトとして統一された導線**で提供する。

---

## 3. サイト全体の設計原則

サイト全体は以下の原則で構成する。

### 3.1 最優先

* ブラウザクラッシュしないこと
* 何のページかすぐ分かること
* 3機能の役割が被らないこと

### 3.2 構造原則

* トップで3機能の違いを明確にする
* 各機能ページは単独で完結する
* 各機能間を相互移動しやすくする
* 説明ページは軽量で静的にする
* 描画ページはCanvas主体で分離する

### 3.3 技術原則

* 静的配信は Cloudflare Pages
* 定期収集は Workers + Cron
* 保存は D1
* 最新キャッシュは KV
* 描画コアは framework-independent
* UIシェルと描画面を分離する

## 5.1 `/` トップページ

トップページは **案内と入口** に徹する。
トップページ自体を重い可視化ページにしない。

### 役割

* Livefield 全体の説明
* 3機能の違いを一目で伝える
* 最新状況の軽い要約を見せる
* 各機能ページへの導線を置く

### 表示要素

* ヒーロー
* 3機能カード
* 現在の観測メタ
* 軽量なサマリー
* Method / About / Status 導線

### やらないこと

* 重いCanvas常時描画
* 全量データをトップで読み込む
* トップで3機能全部をミニ実装すること

トップは軽く、説明的であることを優先する。

## 8. データの流れ

TwitCasting Livefield のデータフローは以下で固定する。

### 8.1 収集

Workers Cron が 1分ごとに TwitCasting 側データを取得する。

### 8.2 生データ保存

取得した raw を D1 に保存する。

### 8.3 ロールアップ

raw から以下を生成する。

* Heatmap 用最新スナップショット
* Day Flow 用 5分ロールアップ
* Battle Lines 用 1分 / 5分 / 10分整形データ

### 8.4 最新キャッシュ

各ページの最新描画に必要な最小JSONだけ KV に保存する。

### 8.5 フロント取得

各ページは対応する `/api/*` から軽量JSONを取得する。

### 8.6 クライアント描画

取得済みJSONを Canvas 2D で描画する。

---

## 9. API 境界

フロントは raw 全量を直接読まない。
API は **ページ単位で必要最小限の整形済みデータ** を返す。

## 9.1 `/api/heatmap`

返すもの:

* 現在表示対象ノード
* summary metrics
* 更新メタ

返さないもの:

* 過去日の全履歴
* 生のコメント一覧
* 全配信者全時刻データ

## 9.2 `/api/day-flow`

返すもの:

* 対象日の Top N + Others
* Volume / Share 描画用系列
* summary
* time focus 用補助

## 9.3 `/api/battle-lines`

返すもの:

* 可視対象配信者の系列
* viewers / indexed 用値
* summary
* focus 用補助

## 9.4 `/api/status`

返すもの:

* last successful update
* stale / partial 判定
* 収集状態の軽量情報

## 10. データ保持方針

### 10.1 D1

D1 は永続本体とする。

保持対象:

* raw snapshots
* 日別ロールアップ
* ページ別整形に必要な中間値

### 10.2 KV

KV は**現在表示に必要な最新キャッシュだけ**に使う。

保持対象:

* latest heatmap payload
* today day-flow latest payload
* today battle-lines latest payload
* status summary

### 10.3 クライアント

クライアントは必要ページの必要データのみ保持する。
全機能分の先読みは禁止。

## 12. レンダラー分離方針

最重要ルールとして、可視化ごとに **専用レンダラー** を持たせる。

### 原則

* Heatmap Renderer
* Day Flow Renderer
* Battle Lines Renderer

は独立実装する。

### 理由

* 各機能の負荷特性が違う
* 片方の変更が他方に影響しにくい
* クラッシュ原因を追いやすい

### 共通化するもの

* Canvas 初期化
* DPR制御
* アニメーションON/OFF
* Low Load Mode
* 共通色管理
* 共通テキスト描画補助

## 13. サイト全体のモード

Livefield 全体で以下の共通モードを持つ。

### 13.1 Normal

通常表示

### 13.2 Animation Off

演出を弱める

### 13.3 Low Load Mode

描画負荷を意図的に落とす

### 13.4 Demo Mode

実データが使えない時の代替表示

これらは**ページごとのローカル設定ではなく、サイト全体に通る共通概念**とする。

## 18. サイト全体の完成条件

TwitCasting Livefield 全体の完成条件は以下とする。

### 必須

* 3機能が役割分担できている
* どのページもクラッシュせず表示できる
* トップから3機能へ自然に遷移できる
* status / method / about が揃っている
* stale / partial / error / demo の状態が分かる
* Low Load Mode が全体で機能する
* モバイルで最低限の主操作が完了する

### 不可

* トップが重すぎる
* 3機能の役割が被る
* 各ページで状態表現がバラバラ
* renderer が密結合で一箇所の変更が全体破壊につながる
* データ取得がページごとに無秩序
* ブラウザクラッシュを放置したまま公開する

---

## 19. サイト全体の一言定義

**TwitCasting Livefield は、TwitCasting の配信空間を “今・今日・比較” の3方向から観測するために、軽量な静的シェルと専用Canvasレンダラーを組み合わせて構成される観測サイトである。**
