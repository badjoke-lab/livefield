# Twitch Livefield サイト全体構成書 v1.2 改訂版

## 1. 文書の位置づけ

本書は、**Twitch Livefield のサイト全体の構造・ページ導線・データの流れ・実装境界**を定める。
優先順位は以下とする。

1. **Twitch Livefield 共通仕様書**
2. **本書（サイト全体構成書）**
3. **Heatmap / Day Flow / Battle Lines 各機能仕様書**
4. 実装タスク

本書は、共通仕様で定めた最上位原則を、**サイト全体のページ構成と導線設計に落とし込む文書**である。
各画面の細かな表示ルールは個別仕様書で扱うが、**どのページが何を担い、どうつながるか** は本書で固定する。

---

## 2. サイト全体の目的

Twitch Livefield は、Twitch のライブ配信空間を以下の3つの視点で観測するサイトとする。

* **Heatmap**
  今この瞬間の勢力図
* **Day Flow**
  1日の支配率と総量の流れ
* **Battle Lines**
  個別比較と逆転・急伸の分析

この3機能を、単発ページの寄せ集めではなく、**1つの観測サイトとして統一された導線**で提供する。

v1 の目的は、まずこの3機能を **Twitch-only MVP** として安定動作させることである。
グローバル競争で勝つための高度機能は、v1 公開後の後続フェーズで扱う。

---

## 3. サイト全体の設計原則

サイト全体は以下の原則で構成する。

### 3.1 最優先

* ブラウザクラッシュしないこと
* 何のページかすぐ分かること
* 3機能の役割が被らないこと

### 3.2 構造原則

* トップで3機能の違いを明確にする
* 各機能ページは単独で完結する
* 各機能間を相互移動しやすくする
* 説明ページは軽量で静的にする
* 描画ページは **機能に応じて最適な方式で分離する**
* Heatmap は production treemap を本番表示とし、bubble は比較用・内部用にのみ残してよい

### 3.3 技術原則

* 静的配信は Cloudflare Pages
* 定期収集は Workers + Cron
* 保存は D1 主体
* KV は使う場合も最小キャッシュに限定
* 描画コアは framework-independent
* UI シェルと描画面を分離する
* 無料運営を前提に、**保存とAPIを無駄なく細く保つ**

### 3.4 v1 の姿勢

* まずは **現行仕様の挙動を Twitch で成立させる**
* 先に広げすぎない
* 勝ち筋の拡張は v1 公開後に行う
* v1 では「世界向け完成版」ではなく「世界向けの土台」を作る

元仕様にあった「トップで違いを明確にする」「各機能を単独完結させる」「説明ページは軽量」という原則は、そのまま維持する。
ただし Heatmap の正式 renderer は bubble field ではなく、production treemap に更新する。

## 5. 各ページの役割

## 5.1 `/` トップページ

トップページは **案内と入口** に徹する。
トップページ自体を重い可視化ページにしない。

### 役割

* Livefield 全体の説明
* 3機能の違いを一目で伝える
* 現在の観測状況の軽い要約を見せる
* 各機能ページへの導線を置く
* Method / About / Donate / Status への導線を置く

### 表示要素

* ヒーロー
* 3機能カード
* 現在の観測メタ
* 軽量サマリー
* Status note
* Support card
* Method / About / Donate / Status 導線

### やらないこと

* 重い可視化の常時描画
* 全量データの読み込み
* 3機能全部のミニ実装
* トップ上での複雑な比較UI
* 長大な説明文の直置き

トップは「軽く」「違いが分かり」「どこへ行けばよいかが分かる」ことを最優先する。

---

## 5.2 `/heatmap`

Heatmap 専用ページ。
役割は **今この瞬間の勢力図**。

### 主目的

* 視聴者数の大きさ
* 配信ごとの現在位置
* 直近の伸び
* 今どこが熱いか

を **production treemap の面積・色・補助演出** で読む。

### 構成

* ヒーロー
* コントロール
* ステータス
* Heatmap treemap board
* 凡例
* 詳細パネル
* 他機能への導線

### このページが担う問い

* 今いちばん大きいのは誰か
* 今伸びているのは誰か
* 今どのタイルが熱いか
* 全体の現在勢力図はどう見えるか

### 補足

* 本番 `/heatmap` は **production treemap** を正式採用する
* 各配信は **tile** として描画する
* 面積は viewers、色は momentum、activity は補助演出として扱う
* 上昇=緑 / 下降=赤 / 横ばい=青灰 を正式ルールとする
* Heatmap treemap は **画面全体を高密度に埋める** ことを MVP 必須条件とする
* zoom は小タイル閲覧のための早期導入対象とする
* grouping は MVP 後でもよいが、将来追加を妨げない設計とする
* bubble renderer を残す場合でも、公開 nav からは外す
* `/heatmap` 上で bubble と treemap を同時比較する UI は置かない
* 1 tile ごとに、選択導線と配信リンク導線の両方を持たせる

## 6. 3機能の役割分離

Livefield の核は、3機能の役割を混ぜないことにある。

### Heatmap がやること

* 今の勢力図
* 面積と色で読む
* 現在の熱量とざわつきを見る

### Day Flow がやること

* 1日の支配率
* 面で読む
* 時間帯ごとの主役を見る

### Battle Lines がやること

* 個別比較
* 線で読む
* 逆転・急伸を読む

### 禁止

* Heatmap で Day Flow 的な1日説明を主役にしない
* Day Flow で Battle Lines 的な線比較を主役にしない
* Battle Lines で Heatmap 的な空間配置を主役にしない
* どのページにも全部入りの万能画面を作らない

サイト全体の強さは、**ページごとの役割差がはっきりしていること** にある。

## 7. データフロー全体像

Twitch Livefield のデータは、以下の流れで扱う。

### 7.1 収集

* Twitch live stream データを 1分ごとに取得
* minute snapshot として記録
* v1 では live 中心の観測を前提とする

### 7.2 保存

* D1 に minute snapshot を保存
* D1 に rollup を保存
* 必要なら最新メタだけを軽量キャッシュする
* 過剰な page-specific 保存は避ける

### 7.3 API 整形

* `/api/heatmap`
* `/api/day-flow`
* `/api/battle-lines`
* `/api/status`

は、D1 由来データを **ページ向けに軽く整形した JSON** として返す。

### 7.4 フロント表示

* フロントは raw を直接読まない
* 各ページはページ用の payload だけを受け取る
* renderer は受け取った page payload を描画する

### 7.5 原則

* 収集と描画を分離する
* API は再集計の場にしない
* ページごとの関心事を分ける
* 1つの変更が全体を壊しにくい構造を保つ

## 8. レイヤー分離

サイト全体は、以下の4層を分離する。

### 8.1 Shell 層

* ナビ
* ヒーロー
* コントロールUI
* ステータス表示
* 説明ページ

### 8.2 Renderer 層

* Heatmap renderer
* Day Flow renderer
* Battle Lines renderer

### 8.3 API 層

* page-oriented JSON を返す
* UIに必要な整形だけを行う

### 8.4 Collector / Storage 層

* 定期収集
* snapshot 保存
* rollup 生成
* status 集計

### 原則

* renderer は他ページ事情を知らない
* shell は重い時系列再計算をしない
* API は描画状態を持たない
* collector は描画都合のコードを持たない

---

## 9. サイト全体の共通モード

Livefield 全体で以下の共通モードを持つ。

### 9.1 Normal

通常表示

### 9.2 Animation Off

演出を弱める

### 9.3 Low Load Mode

描画負荷を意図的に落とす

### 9.4 Demo Mode

実データが使えない時の代替表示

### 9.5 原則

これらは**ページごとのバラバラ設定ではなく、サイト全体に通る共通概念** とする。
どのページでも同じ意味で働かなければならない。

## 10. 共通ページシェル

全ページは、完全に同じレイアウトである必要はないが、共通シェル思想を持つ。

### 必須要素

* ページタイトル
* 一言説明
* current status / last updated
* 共通ナビ
* 他機能への導線
* footer
* モバイルでも崩れない閉じ方

### シェルの目的

* どのページにいても同じサイトだと分かる
* 状態表示の意味が揃う
* 画面遷移が迷子にならない
* renderer 差し替え時に全体の印象が壊れない

## 13. データ範囲と縮退方針

v1 は無料運営前提のため、全ページで観測対象と更新コストに上限を持つ。

### 原則

* 全配信を無制限に扱わない
* まずは上位観測対象で成立させる
* API 制限や保存制限で不足がある場合は partial を出す
* 取得不能な指標は無理に実測値のように出さない

### 許容される縮退

* coverage の限定
* 一部指標の簡略化
* stale note の明示
* demo fallback
* mobile での表示縮小

### 禁止

* 欠損を黙って通常表示に見せる
* 取得できないデータを推測だけで断定表示する
* 無料枠を超える無理な収集設計のまま公開する

---

## 14. API 設計の原則

### 14.1 page-oriented API

API は、ページごとに必要な payload を返す。
フロントに raw の塊を渡さない。

### 14.2 軽量整形

* Heatmap は Heatmap 用 payload
* Day Flow は Day Flow 用 payload
* Battle Lines は Battle Lines 用 payload
* Status は status payload

### 14.3 共通状態

どの API でも最低限以下を返せるようにする。

* data status
* last updated
* coverage
* note
* demo/stale/partial flags

### 14.4 やらないこと

* API で重い再集計を毎回やる
* ページ共通で使わない巨大 payload を返す
* UI専用の一時状態をサーバーで持つ

## 17. サイト全体の完成条件

Twitch Livefield 全体の完成条件は以下とする。

### 必須

* 3機能が役割分担できている
* どのページもクラッシュせず表示できる
* トップから3機能へ自然に遷移できる
* status / method / about が揃っている
* stale / partial / error / demo の状態が分かる
* Low Load Mode が全体で機能する
* モバイルで最低限の主操作が完了する
* Twitch-only MVP として説明が一貫している
* 無料運営前提の収集と表示が継続できる
* `/heatmap` が tile 正式版として読める
* 各 Heatmap タイルから配信導線に辿れる

### 不可

* トップが重すぎる
* 3機能の役割が被る
* 各ページで状態表現がバラバラ
* renderer が密結合で一箇所の変更が全体破壊につながる
* データ取得がページごとに無秩序
* ブラウザクラッシュを放置したまま公開する
* Twitch 前提なのに TwitCasting 前提文言が残る
* 無料枠で継続不能な設計のまま公開する
* `/heatmap` が比較ラボ画面のまま残る
* tile 正式版なのに bubble 依存の説明が残る

## 13. Heatmap production treemap 採用に伴う全体追記事項

本節は、Heatmap 正式版を production treemap として採用したことに伴う、サイト全体構成上の追記事項を定める。

### 13.1 ページ上の位置づけ

* `/heatmap` は **Now view** を担う公開ページである
* `/heatmap` は treemap 1本に絞った本番ページとする
* bubble 比較画面は公開導線から外し、必要なら `/lab/*` に退避する
* `/heatmap` を renderer 比較実験場として扱わない

### 13.2 情報の主従

* viewers は面積で読む
* momentum は色で読む
* activity は補助演出で読む
* sampled / unavailable は隠さず表示する
* ただし partial 表示は主画面の可読性を壊さない位置に整理する

### 13.3 zoom と grouping

* zoom は treemap の正当な補助機能であり、早期導入対象とする
* grouping は MVP 後でもよいが、設計上の前提として許容しておく
* grouping を追加しても Heatmap / Day Flow / Battle Lines の役割差が崩れないようにする

### 13.4 リンク導線

* Heatmap では、気になった配信へ即遷移できることを公開ページの価値とする
* tile 本体は選択
* 明示リンクは配信遷移
* details panel の `Open stream` を必須とする

### 13.5 TradingView 系 treemap を参考にする際の原則

TradingView 型の market heatmap / crypto heatmap は、以下の観点で参照対象として有益である。

* 画面全体を高密度に埋める treemap packing
* size と color の明確な分離
* zoom / tooltip / detail layering
* grouping の後付け容易性

ただし Livefield は Twitch 配信観測サイトであり、固有UIの複製ではなく、**配信観測に合わせて再設計した production treemap** として実装する。
